# الدستور البرمجي للذكاء الاصطناعي (AI_INSTRUCTIONS)

## 🛑 أمر حتمي وصارم: 
**يُمنع منعاً باتاً قراءة، تحليل، أو تعديل أي سطر برمجي في هذا المشروع قبل قراءة وفهم هذا الملف بالكامل. هذا الملف يمثل الدستور البرمجي للمشروع وأي مخالفة لما فيه تُعد إخفاقاً تاماً في أداء المهمة.**

---

## 1. الهيكلة المعمارية للمشروع (Project Architecture)

يعتمد هذا المشروع على إطار عمل Vue 3 مع بيئة تشغيل Vite وإدارة حالة (State Management) باستخدام Pinia. التقسيم المعماري داخل المجلد `src` صارم ويجب الالتزام به كالتالي:

- **واجهات المستخدم (UI):** توضع حصراً في مجلدي `views/` (للصفحات الكاملة) و `components/` (للمكونات القابلة لإعادة الاستخدام). يُمنع وضع أي منطق أعمال (Business Logic) معقد داخل هذه الملفات.
- **إدارة الحالة ومنطق الأعمال (Logic & State):** توضع حصراً داخل مجلد `stores/` باستخدام Pinia. يجب أن تتعامل المكونات مع الـ Stores لاسترجاع البيانات أو تنفيذ الإجراءات.
- **التواصل مع الخوادم (Data & Network):** يتم حصر جميع استدعاءات الشبكة (API Calls) داخل مجلدي `api/` أو `services/`. يُمنع كتابة طلبات الشبكة (Axios وغيرها) مباشرة داخل الـ UI Components.
- **الوظائف المساعدة والقابلة لإعادة الاستخدام:** توضع في مجلد `composables/` (لـ Vue Composition API) أو في `utils/` و `helpers/` (للوظائف العامة المستقلة).
- **التوجيه والتنقل:** يُدار بالكامل عبر مجلد `router/`.
- **الترجمة والنصوص الثابتة:** تدار عبر مجلد `locale/`.

---

## 2. قواعد برمجية صارمة (Strict Coding Guidelines)

يجب على الذكاء الاصطناعي الالتزام التام بالقواعد التالية أثناء توليد أو تعديل الكود:

1. **المنع المطلق للنصوص الثابتة (No Hardcoded Strings):** يُمنع منعاً باتاً كتابة أي نص مقروء للمستخدم النهائي (User-facing text) مباشرة داخل الكود أو قوالب HTML. يجب استخدام نظام الترجمة (i18n) وجلب النصوص من ملفات `locale/`.
2. **الفصل التام للمسؤوليات (No Business Logic in UI):** لا تقم ببرمجة أي عمليات حسابية معقدة، تصفية بيانات متقدمة، أو منطق أعمال داخل ملفات الـ `.vue`. يجب نقل هذا المنطق إلى الـ `stores/` أو الـ `composables/`.
3. **قواعد التسمية (Naming Conventions):**
   - أسماء الملفات في مجلد `components/` يجب أن تكون بنمط PascalCase (مثل `DataTable.vue`).
   - أسماء المتغيرات والدوال يجب أن تكون بنمط camelCase.
   - أسماء الثوابت يجب أن تكون بنمط UPPER_SNAKE_CASE (وتوضع في مجلد `constants/`).
4. **التنظيف والترتيب:** يجب إزالة أي استيرادات (Imports) غير مستخدمة، والالتزام بمعايير تنسيق الكود المعمول بها في المشروع.

## 3. أوامر الطرفية الإلزامية (Mandatory Terminal Commands)

عند تنفيذ أي تعديلات تمس الحزم والمكتبات أو بيئة العمل، يجب تنفيذ الأوامر التالية:

- **عند إضافة أو تعديل مكتبة برمجية:** يجب تشغيل أمر `npm install` أو `npm update`.
- **عند الحاجة لاختبار البناء النهائي:** يجب تشغيل أمر `npm run build` للتحقق من عدم وجود أخطاء تمنع التجميع.
- **عند تشغيل خادم التطوير:** يتم استخدام `npm run dev`.

**أمر صارم ودائم:**
إذا قمت (كذكاء اصطناعي) باتخاذ قرار أو طلب منك إضافة تقنية جديدة، أو مكتبة برمجية ضخمة أساسية، أو تغيير نمط معماري رئيسي في المشروع، **يجب عليك أولاً ومباشرةً** تحديث هذا الملف (`AI_INSTRUCTIONS.md`) لإدراج التقنية الجديدة، أو شرح النمط المعماري الجديد في القسم المخصص له، قبل الشروع في كتابة الكود الخاص بالميزة الجديدة. 

هذا البروتوكول يضمن أن يظل هذا الدستور محدثاً ومرجعاً شاملاً لأي ذيل ذكاء اصطناعي سيقوم بقراءة المشروع في المستقبل.
